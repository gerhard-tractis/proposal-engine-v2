# Tech-Spec: Proposal Engine Refactor ‚Äî HTML-first Architecture

**Created:** 2026-02-15

## Overview

### Problem Statement

The current system requires Vercel redeploys for every component change, is limited to generic React block components, has poor brand color extraction (scrapes UI noise like grays and state colors), and produces visually similar proposals regardless of content. The architecture tightly couples proposal rendering to the Next.js app.

### Solution

Replace React-rendered proposals with self-contained HTML files generated by a 4-agent pipeline (Brand Designer ‚Üí Architect ‚Üí HTML Builder ‚Üí Polish). HTML is stored in Supabase Storage and served through a lightweight Next.js proxy at the existing `/proposals/` route. Tailwind classes are compiled locally before upload. No redeploy needed for new proposals.

### Scope

**In Scope:**
- New DB migration (add `client_name`, `client_url`, `view_count`, `expires_at`, `html_path`, `asset_manifest`; update RLS policies for new status values)
- Supabase Storage buckets (`proposals`, `proposal-assets`) ‚Äî created via Supabase Dashboard/CLI, NOT raw SQL
- `increment_view` DB function
- HTML proxy route handler + old-proposal redirect strategy
- `injectSignedUrls()` utility
- `scripts/generate-proposal.ts` pipeline script with `scripts/package.json`
- 4 agent system prompts (Brand Designer, Architect, HTML Builder, Polish)
- Tailwind CLI standalone compilation step (cross-platform)
- `scrapeBrand()` and `uploadClientAssets()` utilities
- CSP headers for HTML proposal security

**Out of Scope:**
- Removing old React block components (post-migration cleanup)
- Admin dashboard changes
- Deleting old agent prompts (2a, 2b, 3)
- Updating `CLAUDE.md`, `README.md`, `PROJECT_STATE.md`

## Context for Development

### Codebase Patterns

- **Supabase client**: Two-tier ‚Äî `getSupabaseClient()` (anon, Edge-compatible) and `getSupabaseAdmin()` (service role, server-only). Both are lazy singletons in `apps/web/src/lib/supabase.ts`.
- **RLS policies**: Current anon policy is `status = 'published'` only. The middleware uses the anon client (`getSupabaseClient()`), so it is subject to RLS. Any new status values that should be accessible must have RLS policies updated.
- **Zod validation**: Schemas in `packages/shared/src/types/proposal.ts` are single source of truth. Validated on both save (`scripts/save-proposal.ts`) and read (`proposal-helpers.ts`).
- **Next.js App Router**: Async `params` pattern (Next.js 16). `dynamic = 'force-dynamic'`, `revalidate = 0` on proposal pages. **Critical: `route.ts` and `page.tsx` CANNOT coexist in the same directory.**
- **Middleware**: `apps/web/src/middleware.ts` validates tokens at `/proposals/*` using `getSupabaseClient()` (anon key, subject to RLS). Rate limiting (10/min/IP), format validation, fallback env var tokens. Adds `X-Robots-Tag: noindex`.
- **Branding**: `branding.ts` generates 24+ CSS variables from proposal `client.colors`. Hex‚ÜíHSL conversion for Tailwind compatibility. Applied in proposal `layout.tsx`.
- **Proposal layout**: `proposals/[slug]/[token]/layout.tsx` fetches full proposal, applies CSS vars, renders sticky header with logo/title. Supports 3 header styles.
- **Dembrandt**: `dembrandt` is a dependency of `apps/agent` (not globally installed). CLI command extracts hex colors via regex + Puppeteer favicon extraction. Currently in `apps/agent/src/routes/extract-design.ts`.
- **Save flow**: `scripts/save-proposal.ts` reads JSON from CLI arg ‚Üí Zod validates ‚Üí inserts to Supabase via admin client (env from `apps/agent/.env`). Scripts directory has no `package.json`.
- **Monorepo**: Turborepo + pnpm workspaces. `@repo/shared` for types. Build depends on `^build`.
- **No Supabase Storage usage exists** ‚Äî clean slate for bucket setup.
- **Platform**: Windows 11. Shell commands must be cross-platform (no `/dev/null`, use platform-agnostic alternatives).

### Technical Decisions

- **Keep `/proposals/` URL prefix** ‚Äî new proxy lives at same path for consistency
- **Incremental migration** ‚Äî new pipeline runs alongside old system; old React rendering stays until proven
- **Move old page.tsx to `/proposals/[slug]/[token]/legacy/`** ‚Äî since `route.ts` and `page.tsx` cannot coexist, move old rendering to a sub-route. The route handler redirects old proposals (no `html_path`) to `/proposals/{slug}/{token}/legacy` internally.
- **Update RLS policies** ‚Äî add new anon read policy for `status IN ('published', 'sent', 'viewed')` so middleware can validate tokens for HTML proposals without switching to admin client
- **Middleware uses admin client** ‚Äî switch middleware from `getSupabaseClient()` (anon) to `getSupabaseAdmin()` (service role) to bypass RLS entirely. This is simpler than maintaining RLS policies for every new status value. The middleware already runs server-side only.
- **Tailwind CLI v3** ‚Äî use `npx tailwindcss` (NOT `@tailwindcss/cli` which is v4 only). Provide a minimal standalone `tailwind.config.js` with no custom theme (agents use only default Tailwind utilities). Cross-platform: use temp file instead of `/dev/null`.
- **Sonnet 4.5 for all 4 agents** ‚Äî specialized agents with reduced scope, better quality than single Opus pass, ~$0.37/proposal
- **Asset placeholders with manifest** ‚Äî HTML uses `{{logo:client}}`, `{{logo:tractis}}`, `{{favicon:client}}`. Asset paths (with extensions) stored in DB column `asset_manifest JSONB` at upload time, so `injectSignedUrls` knows exact paths without listing buckets.
- **Scripts package.json** ‚Äî create `scripts/package.json` with own dependencies (`@anthropic-ai/sdk`, `nanoid`, `@supabase/supabase-js`, `dotenv`) so scripts are self-contained. Run via `npx tsx scripts/generate-proposal.ts`.
- **Dembrandt via npx** ‚Äî run as `npx --prefix apps/agent dembrandt <url>` to use the version installed in `apps/agent` without global install. Skip Puppeteer favicon extraction in scripts (use Dembrandt output only); favicon can be provided via `--favicon <url>` CLI arg or auto-detected from `<link rel="icon">` via simple HTML fetch + regex (no Puppeteer needed).
- **CSP headers** ‚Äî serve HTML proposals with `Content-Security-Policy: default-src 'self' 'unsafe-inline'; img-src * data:; font-src *` to prevent malicious script injection from LLM-generated content.
- **LangChain for agent calls** ‚Äî use existing LangChain + `@langchain/anthropic` (already in `apps/agent`) instead of introducing a second AI SDK. Import from `apps/agent` or duplicate the dependency in `scripts/package.json`.

#### Phase 2: HTML Proxy Route

- [ ] Task 6: Move old proposal rendering to legacy sub-route
  - Files:
    - Move `apps/web/src/app/proposals/[slug]/[token]/page.tsx` ‚Üí `apps/web/src/app/proposals/[slug]/[token]/legacy/page.tsx`
    - Move `apps/web/src/app/proposals/[slug]/[token]/layout.tsx` ‚Üí `apps/web/src/app/proposals/[slug]/[token]/legacy/layout.tsx`

  - Notes: This is the cleanest way to handle the Next.js constraint that `route.ts` and `page.tsx` cannot coexist. Old proposals get internally redirected to `/proposals/{slug}/{token}/legacy` by the route handler (invisible to users).

- [ ] Task 7: Create HTML proxy route handler
  - File: `apps/web/src/app/proposals/[slug]/[token]/route.ts`
  - Action: Create GET route handler that:
    1. Receives `slug` and `token` from async params
    2. Queries `proposals` table via admin client for matching row (select `html_path`, `expires_at`, `asset_manifest`)
    3. **If row not found** ‚Üí return 404
    4. **If `html_path` is NULL** (old React proposal) ‚Üí return `NextResponse.rewrite(new URL(\`/proposals/${slug}/${token}/legacy\`, request.url))` to render via old React page
    5. **If `html_path` is set** (new HTML proposal):
       a. Check `expires_at` ‚Äî if past, return `new Response('Proposal expired', { status: 410 })`
       b. Call `increment_view` RPC via `supabase.rpc('increment_view', { proposal_slug: slug })`
       c. Fetch HTML from Storage: `supabase.storage.from('proposals').download(html_path)`
       d. Call `injectSignedUrls(html, slug, assetManifest)`
       e. Return `new Response(html, { status: 200, headers: { 'Content-Type': 'text/html; charset=utf-8', 'X-Robots-Tag': 'noindex, nofollow', 'Cache-Control': 'private, no-cache', 'Content-Security-Policy': "default-src 'self' 'unsafe-inline'; img-src * data:; font-src *" } })`
  - Notes: The `NextResponse.rewrite()` for old proposals is an internal rewrite ‚Äî the user's URL stays the same, but Next.js renders the `legacy/` route. This avoids reimplementing the entire React rendering pipeline inside the route handler. Export `dynamic = 'force-dynamic'` to prevent caching.

- [ ] Task 8: Create `injectSignedUrls` utility
  - File: `apps/web/src/lib/proposal-helpers.ts`
  - Action: Add function `injectSignedUrls(html: string, slug: string, assetManifest: AssetManifest): Promise<string>` that:
    1. Creates Supabase admin client
    2. For each placeholder, use the **exact path from `assetManifest`** (not guessed):
       - `{{logo:client}}` ‚Üí `supabase.storage.from('proposal-assets').createSignedUrl(assetManifest.clientLogo, 3600)`
       - `{{logo:tractis}}` ‚Üí `supabase.storage.from('proposal-assets').createSignedUrl(assetManifest.tractisLogo, 3600)`
       - `{{favicon:client}}` ‚Üí if `assetManifest.clientFavicon` exists, create signed URL; otherwise remove the placeholder
    3. Replace placeholders in HTML string using exact string match (only `{{logo:client}}`, `{{logo:tractis}}`, `{{favicon:client}}` ‚Äî these are intentionally distinct patterns unlikely to appear in content)
    4. Return processed HTML
  - Notes: Using the asset manifest eliminates the guessing problem (F13). Extensions are stored at upload time and reused at serve time.

- [ ] Task 9: Add `getHtmlProposal` helper
  - File: `apps/web/src/lib/proposal-helpers.ts`
  - Action: Add function `getHtmlProposal(slug: string, token: string): Promise<{ html: string; expired: boolean; assetManifest: AssetManifest } | null>` that:
    1. Queries `proposals` table via admin client for row matching slug + token where `html_path IS NOT NULL`
    2. Checks `expires_at` ‚Äî if past, returns `{ html: '', expired: true, assetManifest: {} }`
    3. Calls `increment_view` RPC
    4. Fetches HTML from Storage bucket using `html_path`
    5. Returns `{ html, expired: false, assetManifest }` (signed URL injection happens in route handler)
  - Notes: Keeps existing `getProposalBySlugAndToken` untouched for old proposals. Does NOT call `injectSignedUrls` ‚Äî that happens in the route handler so the asset manifest can be passed through.

- [ ] Task 11: Create `scrapeBrand` utility function
  - File: `scripts/lib/brand.ts`
  - Action: Create function `scrapeBrand(url: string): Promise<DembrandtResult>` that:
    1. Runs Dembrandt via `npx --prefix ../apps/agent dembrandt <url>` using `child_process.execSync` (60s timeout, 10MB buffer). This uses the version installed in `apps/agent` without requiring global installation.
    2. Parses stdout for hex colors via regex `/#([0-9a-fA-F]{6})/g`, deduplicates
    3. Extracts favicon URL via simple HTTP fetch of the URL + regex match on `<link[^>]*rel=["'](?:shortcut )?icon["'][^>]*href=["']([^"']+)["']` ‚Äî NO Puppeteer needed (avoids 400MB Chromium dependency)
    4. Returns `{ colors: string[], favicon: string | null, raw: string }`
  - Notes: Favicon extraction uses a simple `fetch()` + HTML regex instead of Puppeteer. This covers 95% of cases. For sites with JS-rendered favicons, the `--favicon <url>` CLI override in the main script serves as fallback.

- [ ] Task 12: Create `uploadClientAssets` utility function
  - File: `scripts/lib/assets.ts`
  - Action: Create function `uploadClientAssets(slug: string, logoUrl: string, faviconUrl: string | null): Promise<AssetManifest>` that:
    1. Downloads logo from URL using `fetch`
    2. Determines extension from Content-Type header (e.g., `image/svg+xml` ‚Üí `svg`, `image/png` ‚Üí `png`)
    3. Uploads to `proposal-assets/{slug}/logo.{ext}` via Supabase Storage
    4. If favicon URL exists, downloads and uploads to `proposal-assets/{slug}/favicon.{ext}`
    5. Ensures Tractis logo exists at `proposal-assets/tractis/logo.svg` (check existence first, upload only if missing)
    6. **Returns `AssetManifest`** object: `{ clientLogo: '{slug}/logo.{ext}', clientFavicon: '{slug}/favicon.{ext}' | undefined, tractisLogo: 'tractis/logo.svg' }`
  - Notes: The returned manifest is stored in the DB `asset_manifest` column so `injectSignedUrls` knows exact paths. Uses `supabase.storage.from('proposal-assets').upload(path, buffer, { contentType, upsert: true })`.

- [ ] Task 13: Create `compileTailwind` utility function
  - File: `scripts/lib/tailwind.ts`
  - Action: Create function `compileTailwind(htmlContent: string): Promise<string>` that:
    1. Writes HTML to a temp file in `os.tmpdir()` (cross-platform)
    2. Creates a minimal empty CSS input file in `os.tmpdir()` (cross-platform replacement for `/dev/null`):
       ```
       @tailwind base;
       @tailwind components;
       @tailwind utilities;
       ```
    3. Runs **`npx tailwindcss -i <inputCssFile> --content <tempHtmlFile> --minify`** (NOT `@tailwindcss/cli` which is v4 only)
    4. Captures compiled CSS output (write to another temp file via `-o <outputFile>`, then read it)
    5. In the HTML: removes any `<script src="...tailwind...">` or `<script src="https://cdn.tailwindcss.com...">` tags via regex
    6. Injects compiled CSS as `<style>` block inside `<head>` (before `</head>`)
    7. Cleans up all temp files
    8. Returns self-contained HTML
  - Notes: Uses `npx tailwindcss` (Tailwind v3 CLI). No custom tailwind.config needed ‚Äî agents must use ONLY default Tailwind utility classes (no custom theme extensions). The agent prompts (Tasks 15-18) must explicitly state this constraint. All custom styling should use inline CSS custom properties on `:root`, not Tailwind config extensions.

- [ ] Task 14: Create main `generate-proposal.ts` pipeline script
  - File: `scripts/generate-proposal.ts`
  - Action: Create CLI script that orchestrates the full pipeline:
    1. Parse CLI args: `--doc <path>` (proposal markdown), `--client <name>`, `--url <website>`, `--logo <url>` (optional override), `--favicon <url>` (optional override)
    2. Load env from `../apps/agent/.env` via dotenv (same pattern as `save-proposal.ts`)
    3. Generate slug from client name (lowercase, replace spaces with hyphens, strip non-alphanumeric)
    4. Generate token via `nanoid(10)`
    5. **Step 1**: `scrapeBrand(url)` ‚Üí raw brand data. **On failure**: log warning, continue with empty colors (Agent 1 handles gracefully)
    6. **Step 2**: `uploadClientAssets(slug, logoUrl || extracted, faviconUrl || extracted)` ‚Üí asset manifest
    7. **Step 3**: Call Agent 1 (Brand Designer) via Anthropic SDK ‚Äî input: raw Dembrandt JSON ‚Üí output: design system JSON. **Validate** output against `DesignSystemSchema`. **On failure**: retry once, then abort with error.
    8. **Step 4**: Call Agent 2 (Architect) via Anthropic SDK ‚Äî input: design system + proposal markdown ‚Üí output: section architecture JSON. **Validate** against `SectionBlueprintSchema` array. **On failure**: retry once, then abort.
    9. **Step 5**: Call Agent 3 (HTML Builder) via Anthropic SDK ‚Äî input: architecture + design system + markdown ‚Üí output: raw HTML. Set `max_tokens: 16384`. **Check for truncation**: if response `stop_reason !== 'end_turn'`, log warning. **On failure**: retry once, then abort.
    10. **Step 6**: Call Agent 4 (Polish) via Anthropic SDK ‚Äî input: raw HTML ‚Üí output: polished HTML. Set `max_tokens: 16384`. **On failure**: skip polish, use Agent 3 output (polish is optional improvement, not critical).
    11. **Step 7**: `compileTailwind(polishedHtml)` ‚Üí self-contained HTML
    12. **Step 8**: Upload HTML to `proposals/{slug}.html` in Supabase Storage
    13. **Step 9**: Insert record in `proposals` table: `slug`, `token`, `client_name`, `client_url`, `status: 'draft'`, `html_path: '{slug}.html'`, `asset_manifest`
    14. **Step 10**: Print URL `https://proposal.tractis.ai/proposals/{slug}/{token}`
  - Notes: Each agent call reads its system prompt from `apps/agent/prompts/` via `fs.readFileSync`. Use `@anthropic-ai/sdk` directly (from `scripts/package.json`). Log progress for each step to stdout. Wrap entire pipeline in try/catch ‚Äî on any unrecoverable failure, clean up uploaded assets and print clear error message.

### Notes

**High-risk items:**
- **Agent output quality** ‚Äî Agents may produce HTML that doesn't match the design system or has visual bugs. Agent 4 (Polish) mitigates this but isn't guaranteed. Manual review of first 5-10 proposals is critical.
- **Tailwind compilation** ‚Äî Agents must use ONLY default Tailwind utility classes. If they invent custom classes (e.g., `text-brand-primary`), those will silently disappear during compilation. The agent prompts explicitly forbid this, but LLMs may not always comply. Visual review is essential.
- **Large HTML output** ‚Äî Agent 3 may produce very large HTML (2000+ lines). `max_tokens: 16384` allows ~12,000 tokens of HTML output. If truncated (`stop_reason !== 'end_turn'`), the pipeline logs a warning. For very long proposals, content may need to be split across multiple agent calls (out of scope for v1).
- **View count inflation (F9)** ‚Äî `increment_view` fires on every request including page refreshes. This is acceptable for v1 ‚Äî the count represents "page loads" not "unique views." If unique views are needed later, add session-based deduplication (out of scope).
- **HTML security (F18)** ‚Äî LLM-generated HTML is served directly. CSP headers (`default-src 'self' 'unsafe-inline'`) prevent loading external scripts but allow inline styles/scripts (needed for animations). Proposal markdown comes from Gerhard (trusted), not external users, so prompt injection risk is low. If external markdown sources are added later, HTML sanitization should be revisited.
